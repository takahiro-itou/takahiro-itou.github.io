#  本記事の内容

別記事 "補数と引き算" (2025/10/01) にある Question の解答例を示す

##  Question 1

```
やはり引き算が残っているように見えるが、
10000 から引くより 9999 から引く方が
（コンピュータにとっては）遥かに楽である。
それは何故か？
```

先ほどまでの引き算と根本的に違うのは、
引かれる数の全ての位が最大の９になっている。
この引き算では繰り下がりが決して発生しない。

つまり、それぞれの位ごとに引き算ができればよい。

各ビット（各位）毎に演算を適用するというのは、
繰り上がりや繰り下がりで他の位の計算結果に依存する演算に比べてコストが低い。

それでも９から引くという計算が残っているがこれは、単純に各位毎に
$$  0 \leftrightarrow 9, 1 \leftrightarrow 8, \ldots, $$
というような変換を行うだけで良いから、演算コストはそれほど高くない。

##  Question 2
```
Question 2
二進数の計算においては
11111111 - 00011100 は引き算をせずに計算することができる。
どうすればよいか？
```

まず Question1 と同じ理由によって、繰り下がりが発生しないことに注意する。
すると、各ビット（各位）ごとに
$$1 - x$$ という計算ができればよい。

しかし二進数では $x$ は 0 か 1 しかないので、
$$1 - 0 = 1$$ か $$1 - 1 = 0$$
という計算だけができればよい。

[以前の記事]({{ site.baseurl }}{% post_url 2024/2024-04-16-bitwise-operators %})
で、ビット演算 (Bitwise Operators)  を解説しているので、
そちらも読んでおいてほしいが、

$$x$$ の 0 or 1 を反転される

| $$x$$ | $$ \lnot x $$|
|:-----:|:------------:|
|     0 | 1 |
|     1 | 0 |

でよく、これを各ビット毎に適用すればよい。
C 言語では ~x と書ける。

もしくは 定数 1 と $$x$$ の xor をとっても 0 or 1 を反転できるから
$$11111111 xor x$$ であり C 語風に書けば 0b11111111 ^ x となる。
