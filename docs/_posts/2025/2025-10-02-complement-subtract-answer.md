#  本記事の内容

別記事 "補数と引き算" (2025/10/01) にある Question の解答例を示す

##  Question 1

```
やはり引き算が残っているように見えるが、
10000 から引くより 9999 から引く方が
（コンピュータにとっては）遥かに楽である。
それは何故か？
```

先ほどまでの引き算と根本的に違うのは、
引かれる数の全ての位が最大の９になっている。
この引き算では繰り下がりが決して発生しない。

つまり、それぞれの位ごとに引き算ができればよい。

各ビット（各位）毎に演算を適用するというのは、
繰り上がりや繰り下がりで他の位の計算結果に依存する演算に比べてコストが低い。

それでも９から引くという計算が残っているがこれは、単純に各位毎に
$$  0 \leftrightarrow 9, 1 \leftrightarrow 8, \ldots, $$
というような変換を行うだけで良いから、演算コストはそれほど高くない。

##  Question 2
```
Question 2
二進数の計算においては
11111111 - 00011100 は引き算をせずに計算することができる。
どうすればよいか？
```

まず Question1 と同じ理由によって、繰り下がりが発生しないことに注意する。
すると、各ビット（各位）ごとに
$$1 - x$$ という計算ができればよい。

しかし二進数では $$x$$ は 0 か 1 しかないので、
$$1 - 0 = 1$$ か $$1 - 1 = 0$$
という計算だけができればよい。

[以前の記事]({{ site.baseurl }}{% post_url 2024/2024-04-16-bitwise-operators %})
で、ビット演算 (Bitwise Operators)  を解説しているので、
そちらも読んでおいてほしいが、

$$x$$ の 0 or 1 を反転させる

| $$x$$ | $$ \lnot x $$|
|:-----:|:------------:|
|     0 | 1 |
|     1 | 0 |

でよく、これを各ビット毎に適用すればよい。
C 言語では ~x と書ける。

もしくは 定数 1 と $$x$$ の xor をとっても 0 or 1 を反転できるから
`11111111 xor x` であり C 語風に書けば `0b11111111 ^ x` となる。

##  Question3

| CPU      | 繰り下がり無し | 繰り下がり有り |
|:---------|:--------------:|:--------------:|
| 6502など | CF == 1        | CF == 0        |

```
a - b - 繰り下がり

この式をさらに簡単にしなさい。
具体的には変数 a, b, CF を用いて表しなさい。

ヒント：最終的な項の数は 3 以下
```

解答

まず、今まで議論したように -b を補数を使って書き直すと
$$a - b = a + ~b + 1$$ であった。

これに繰り下がりを引いて
$$a - b - 繰り下がり = a + ~b + 1 - 繰り下がり$$

補数計算の時に出ていた + 1 をするのをちょっと待っておいて、
ここにさらに繰り下がりがあれば 1 を引くわけであるが、
補数計算で出てきた +1 と合わせて、1 - 1 で 0 を足せばよい。

逆に繰り下がりがなければ補数の +1 をそのまま足せばよい。

ここで繰り下がりの有無とキャリーフラグ CF の値をもう一度まとめると、

| 繰り下がり | 1 - 繰り下がり | CF の値 |
|:----------:|:--------------:|:-------:|
| 無し (0)   | 1              | 1       |
| 有り (1)   | 0              | 0       |

となって、うまいこと 1 - 繰り下がりと CF が一致している。
したがって最終的な解答は

$$a + ~b + CF$$

となる。
